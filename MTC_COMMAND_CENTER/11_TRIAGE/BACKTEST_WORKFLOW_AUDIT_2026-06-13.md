# Backtest / Optimization Workflow Audit - 2026-06-13

## 1. Executive Summary

This is a read-only audit. No backtest was started, no existing repo file was modified, and `MTC_V2.pine` was not touched.

The current workflow is strong on statistical gates after a run has executed: rolling walk-forward, lockbox OOS, buy-and-hold, CPCV, PBO, evaluation artifacts, Gate scorecards, scorecard ingestion, heartbeat display, and morning reports already exist. The weak part is launch governance. Large runs can still be planned in an AI prompt or shell script and launched before a durable, UI-visible, user-approved run plan exists.

Main conclusion: update schema first, then backend readers/API, then UI, then docs/prompts. The launch artifacts need to become a contract before docs and prompts tell agents to rely on them.

Top architecture gaps:

1. No `run_plan.json` or `run_plan.md` artifact exists.
2. No approval state is modeled for backtest launches.
3. QuantLens research backtests do not expose an explicit backtest profile (`SOURCE_NAKED`, `RISK_NORMALIZED`, `MTC_LIGHT`, `FULL_MTC_CANDIDATE`).
4. QuantLens sizing is normalized trade-return/R-based, not explicit account-risk notional sizing; leverage is not calculated or capped there.
5. Dashboard reads results and heartbeat, but not pre-run plans, approval, parameter rationale, sizing assumptions, or profile-separated results.
6. Artifact discovery is filename-based and partial; no canonical `artifact_index.json` exists.
7. Prompt/runbook still allow AI-supervised launch after plan/smoke; they do not require UI approval before full execution.

Verified current snapshot anchors:

- Dashboard snapshot: 837 scorecards, 19 scorecard runs, 176 pipeline rows, 176 audit rows, 80 backtest runs.
- Required target artifact names found: `run_plan.json` 0, `run_plan.md` 0, `progress.json` 0, `artifact_index.json` 0, `backtest_results.json` 0. Legacy `run_status.json` exists only in 3 older detached locations. `MORNING_REPORT.md` exists in current run folders.
- Current heartbeat reader scans `_heartbeat*.json`, but the latest dashboard heartbeat was unavailable at audit time.

## 2. Current Workflow Map

Current candidate-to-dashboard flow:

1. Source intake and strategy folders live under `03_QUANTLENS/strategies/*` with `producer_spec.json`, `07_deterministic_spec.md`, metadata, and source-intake artifacts.
2. Registries under `05_REGISTRY/*.json` and generated dashboard readers expose strategy/candidate state.
3. `audit_reader.py` computes `eligible_for_backtest` from source material, deterministic rules, duplicate status, blocked reasons, and current stage.
4. `build_needs_backtest_selector.py` writes objective candidates where `eligible_for_backtest == true`, no `scorecard_v2` exists, and the QuantLens expert verdict is not a hard stop.
5. Actual selection for sprint/overnight runs is still script/prompt driven. Examples: `overnight_v2_runner.py`, `strat_extra_runner.py`, dated loop scripts, and `night_runner.sh`.
6. `mega_walk_forward.py` executes strategy x symbol x timeframe x parameter-grid cells, validates data availability, classifies results, and writes `MEGA_walk_forward_results.json`, checkpoint, partial JSON, and report.
7. Validation tail runs `cpcv_validator.py`, `probabilistic_pbo.py`, `build_evaluation_artifact.py`, `score_gate2.py`, `build_all_gate_evidence.py`, `enrich_gate3_evidence.py`, `score_gate3.py`, `score_all_gates.py`, and `alpha_vs_buyhold.py`.
8. Morning reports are generated by `generate_morning_report.py`, `write_overnight_morning_report.py`, or run-specific report helpers.
9. Dashboard readers ingest `*_results.json`, nested `MEGA_walk_forward_results.json`, older detached `run_status.json`, `scorecard_v2/*.scorecard.json`, and `_heartbeat*.json`.
10. `read_model.py` assembles the read-only dashboard snapshot with `backtest_status`, `scorecards`, `candidate_pipeline`, `candidate_audit`, `overnight_heartbeat`, `strategy_research`, and related domains.

Current long-run launch flow:

- Gate 0 docs require pre-read.
- Prompt/runbook asks the agent to define scope, plan, smoke, launch background loop, monitor, run validation tail, generate report, update lessons/handoff.
- The plan is conversational and script-local. It is not a durable artifact with user approval state.

## 3. Current Position Sizing and Leverage Behavior

QuantLens research engine (`03_QUANTLENS/tools/mega_walk_forward.py`):

- Uses signal, next-bar open entry, stop, target, and exit simulation.
- Computes raw percent return from entry/exit price, subtracts `COST_BPS = 8.0`, compounds per-trade returns, and computes R from stop distance.
- Applies a separate post-hoc slippage stress via `SLIPPAGE_BPS_PER_SIDE = 2.0`.
- Uses stop distance for target/R metrics, but does not calculate account-dollar quantity, notional, margin, implied leverage, or max leverage cap.
- No explicit initial capital is used in QuantLens metrics; equity curves are normalized to `1.0`.
- Current report fields are therefore not directly comparable to risk-normalized account sizing unless every run is labeled as normalized source/backtest return.

MTC-Light / MTCRunner path (`02_MTC_BACKTEST`):

- `src/config/profiles/light_risk.py` turns filters/guards off and risk on.
- Defaults: risk long 4%, risk short 3%, max leverage cap 5x, fallback quantity 5%.
- `src/modules/risk/position_sizer.py` implements risk amount, stop-distance sizing, leverage cap, fallback sizing, explicit leverage sizing, and effective leverage.
- `src/cli/mtc_engine_validate.py` runs a manual producer adapter through the existing `MTCRunner`, reports buy-and-hold, excess alpha, risk settings, and artifact paths.
- This is the closest existing path to the proposed `MTC_LIGHT` profile, but it is shortlist-only and not wired into the main Launch Center model.

Current answer to the audit questions:

| Question | Current behavior |
|---|---|
| If a source has no risk management, does current QuantLens use full equity? | No explicit full-equity account model was found. It uses normalized per-trade price returns and compounds them. |
| Does QuantLens use leverage? | No explicit leverage calculation/cap in `mega_walk_forward.py`. |
| Does it calculate implied leverage? | No. |
| Does it cap leverage? | Not in QuantLens; yes in MTC position sizing. |
| Is sizing strategy-specific or global? | QuantLens uses global normalized simulation plus strategy-specific stops/exits. MTC-Light uses global risk profile plus optional overrides. |
| Are fees/slippage/spread applied? | QuantLens applies cost bps and slippage stress; spread is not explicit. MTC fee model has commission and slippage ticks. |
| Are results comparable across stop sizes? | Only as normalized strategy returns, not as equal account-risk returns. Stop-width differences are not converted into position size in QuantLens. |
| Are sizing assumptions shown in reports/UI? | Partially for costs/metrics; not enough for profile, leverage, position sizing, or missing source risk management. |

## 4. Current Strategy Selection Logic

Eligibility exists; launch selection is still mostly manual or script-driven.

Key current logic:

- `audit_reader.py` marks a row eligible when it is not duplicate/blocked, has source material, has deterministic rules, or is already backtested/promoted.
- `build_needs_backtest_selector.py` selects eligible rows with no `scorecard_v2` and filters out SALVAGE/GARBAGE/CLOSED_SOURCE_STOP expert verdicts.
- `pipeline_reader.py` represents pipeline stages and recommended next actions such as "Run backtest".
- Dated run scripts and runners decide actual strategy universe by importing/patching `GRIDS`, selecting runner files, and passing CLI filters.

Gap: there is no durable selector output -> run plan -> UI approval -> execution chain. The selector can produce candidates, but the chosen subset, exclusion reasons, symbols, timeframes, parameter grids, cell count, and sizing profile are not captured as one approval artifact.

## 5. Current Symbol / Timeframe / Parameter Space Logic

Current defaults and mechanisms:

- `mega_walk_forward.py` defines default crypto `SYMBOLS` and `TIMEFRAMES = ["15m", "1h", "2h", "4h", "1D"]`.
- Strategy parameter grids live in `GRIDS` inside `mega_walk_forward.py`, then are extended by `overnight_v2_runner.py` and `strat_extra_runner.py`.
- CLI filters support narrowed strategy/symbol/timeframe runs.
- Grid size/cell count is derived from number of strategies, symbols, timeframes, and per-strategy grid lengths; launch scripts sometimes estimate this manually.
- Data ranges are validated through manifest lookup, row count, `MIN_BARS_REQUIRED`, fold feasibility, and `data_check.py` for single-strategy workflow.
- Missing data and insufficient history become `NO_DATA`, `INSUFFICIENT_DATA`, or `INSUFFICIENT_TRADES`.

Gap: parameter reasons are embedded in code or strategy specs, not surfaced in a launch plan. A user cannot see "why this parameter is optimized" or "why this symbol/timeframe was selected" before execution.

## 6. Current Execution and Monitoring Flow

Current execution:

- `night_runner.sh` is the generic chain: smoke gate -> sweep -> `mcc_night_tail.sh` -> release.
- Dated loop scripts such as `overnight_loop_2026-06-08.sh` implement worker count, run directories, heartbeat, deadline, crash handling, best-iteration selection, validation tail, alpha, and morning report.
- `single_strategy_backtest.py` wraps data validation, MEGA run, CPCV, PBO, summary, and log for one strategy/symbol/timeframe.
- `mcc_night_tail.sh` enriches existing MEGA result dirs into CPCV/PBO/evaluation artifacts/Gate2/Gate3/scorecard_v2/alpha/morning report.

Current monitoring:

- `_heartbeat*.json` files in `03_QUANTLENS/tools/overnight_runs/`.
- `heartbeat_reader.py` chooses newest heartbeat and exposes availability, age, alive/stale state, run id, stage, status, and raw payload.
- `app.js` renders an overnight worker monitor card.

Gap: monitoring is heartbeat-only, not plan-aware. No `progress.json` contract exists for current cell counts, current strategy, current symbol/timeframe, errors, skipped cells, smoke result, or planned-vs-actual progress.

Gap: artifacts are post-run evidence. There is no pre-run plan preview, approval record, plan version, editable user decision, profile-specific result separation, or canonical artifact index.

## 8. New Target Architecture

Target workflow:

1. Planner creates `run_plan.json` and `run_plan.md`.
2. Backend validates the plan and exposes it read-only to the UI.
3. Backtest Launch Center shows strategy selection, exclusion reasons, symbols, timeframes, data coverage, parameter space, profile, sizing/leverage assumptions, cell/runtime estimates, workers, output dir, validation gates, and smoke command.
4. User approves or edits the plan outside the read-only dashboard write boundary. If dashboard writes remain permanently disallowed, approval must be represented by a file produced by a controlled local CLI or manual drop, not by a dashboard write-back.
5. Smoke test runs only after approval.
6. Local engine writes `run_status.json`, `progress.json`, `heartbeat.json`, `artifact_index.json`, and results.
7. UI reads artifacts; AI comments only on request.
8. Results attach back to Strategy Intelligence Page v2 and Backtest Result Explorer.

Important constraint: current pickup notes say the dashboard is permanently read-only. Therefore "user approves from UI" conflicts with the existing hard constraint unless approval is implemented as display plus a separate local approval mechanism. Recommended interpretation: UI displays the approval package; the approval file is created by a controlled CLI/manual action outside the read-only dashboard API.

## 11. Proposed New / Updated Schemas

New schemas:

- `run_plan.schema.json`
  - `run_id`, `created_at`, `created_by`, `run_type`, `execution_mode`
  - `profile`: `SOURCE_NAKED`, `RISK_NORMALIZED`, `MTC_LIGHT`, `FULL_MTC_CANDIDATE`
  - `strategy_selection`: selected/excluded strategies with reasons
  - `symbols`, `timeframes`, `data_coverage`
  - `parameter_space`: per strategy parameter names, values, reason, grid count
  - `sizing`: default model, risk pct, leverage assumption, max leverage, missing source risk note
  - `costs`: fees, slippage, spread assumptions
  - `estimates`: cells, runtime, workers
  - `validation_gates`
  - `approval`: status, approver, timestamp, approval source, notes
  - `output`: run dir, artifact paths

- `run_status.schema.json`
  - `run_id`, `plan_id`, `status`, `stage`, `started_at`, `ended_at`, `last_update_at`
  - `smoke_status`, `approval_status`, `exit_code`, `error_summary`
  - `token_free_execution`: boolean and explanation

- `progress.schema.json`
  - `total_cells`, `completed_cells`, `skipped_cells`, `failed_cells`
  - `current_strategy`, `current_symbol`, `current_timeframe`
  - `current_stage`, `eta_seconds`, `workers`, `last_log_excerpt`

- `artifact_index.schema.json`
  - typed artifacts with `key`, `path`, `kind`, `exists`, `schema`, `producer_step`, `created_at`

- `backtest_profile_result.schema.json`
  - `profile`, `strategy_id`, `symbol`, `timeframe`, `result_kind`
  - `sizing_assumptions`, `leverage_assumptions`, `cost_assumptions`
  - `metrics`, `benchmark`, `robustness`, `promotion_recommendation`

Update schemas:

- `backtest_status.schema.json`: make run status less permissive or reference `run_status`.
- `evaluation_artifact_v1.schema.json`: add or reference profile/sizing/leverage metadata.
- `production_readiness_artifact_v1.schema.json`: cross-link `FULL_MTC_CANDIDATE` readiness fields.
- `strategy_registry.schema.json` / `research_backtest_registry.schema.json`: optional latest plan/result references.

## 15. Risks and Open Questions

Risks:

- Dashboard write-back conflict: current hard constraint says read-only dashboard permanently. Backtest Launch Center approval must not violate that without explicit owner decision.
- Trading logic boundary: adding profile labels and artifact writers is safe; changing sizing behavior is trading-logic-adjacent and needs approval.
- Historical result interpretation: old results cannot be retroactively treated as RISK_NORMALIZED or MTC_LIGHT unless their artifacts prove it.
- Profile sprawl: do not build a new engine. Use current QuantLens and `MTCRunner` paths.
- False comparability: UI must prevent SOURCE_NAKED vs MTC_LIGHT direct comparison.

Open questions:

1. Should approval be represented by a manually dropped JSON file, a CLI command, or reopening limited dashboard write-back?
2. What default risk pct and max leverage should `RISK_NORMALIZED` use?
3. Should `SOURCE_NAKED` default be explicitly "normalized 1x no leverage" or "normalized return, not account-sized"?
4. Which run types require approval? Recommended: sprint/overnight/all multi-strategy runs; optional for single-strategy smoke.
5. Where should plans live? Recommended: `03_QUANTLENS/05_BACKTEST_RESULTS/plans/<run_id>/` or `03_QUANTLENS/run_plans/<run_id>/`.
